# W-09 — 비병합 상태에서 Slack 고려를 docs-only/flag/bridge 중 어떻게 반영할지

## 문맥
- 요구사항이 “지금 합치지 않음”이므로, 실행 수준을 명확히 못 박아야 계획이 산으로 가지 않는다.

## 선택지별 트레이드오프
### A) docs-only 계약 반영
- 트레이드오프: 범위 통제는 강하지만 코드 검증이 부족할 수 있다.
- 되돌리기 비용(코드 변경량): 10줄 미만

### B) feature flag 뒤 어댑터 skeleton
- 트레이드오프: 실행 가능성을 확인할 수 있지만 초기 코드 부피가 늘어난다.
- 되돌리기 비용(코드 변경량): 300줄 미만

### C) dual-runtime bridge
- 트레이드오프: 통합 가능성은 높지만 현재 목표 대비 위험과 복잡도가 과하다.
- 되돌리기 비용(코드 변경량): 그 이상

## 질문 중요도
- 최대 되돌리기 비용: 그 이상
- 중요도: 최상 (되돌릴 때 바뀌는 코드 경로의 폭 기준)

## 결정 체크포인트
- 선택 즉시 영향받는 영역: 식별자 스키마, 세션 키, 권한/저장소 partition, 채널 어댑터 계약
- 결정 후 확인할 지표: 세션 충돌 건수, 권한 오판률, adapter 변경량, cross-repo 드리프트 건수
